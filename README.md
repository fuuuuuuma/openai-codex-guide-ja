## 1. そもそも Codex とは（30秒で）

Codex（コーデックス）は、**OpenAI が出している「コーディング用のAIエージェント」**です。

ChatGPT が「文章で答えてくれるAI」だとすれば、Codex は **「あなたのコードやPCに手を入れて、実際に作業まで進めてくれるAI」** だと考えると分かりやすいです。

公式の説明では、Codex は **ChatGPT の有料プラン、または OpenAI API のクレジット**を通じて使えるAIコーディングアシスタントで、

- コードの生成
- バグの発見と修正
- プロジェクト内のタスク処理

などを、いくつもの入り口（後述の4つの入り口＋スマホ遠隔）から手伝ってくれる、とされています（出典: 公式 quickstart）。

ポイントは、ただ提案するだけでなく **「実際にファイルを書き換えるところまでやる」** こと。だからこそ公式も、作業を任せる前に **Git でチェックポイント（コミット）を取っておくこと**を推奨しています。これは「AIに任せた結果が気に入らなければ、いつでも前の状態に戻せるようにしておこう」という安全策です（出典: 公式 quickstart）。

> 💡 **たとえ話**: Codex は「すごく優秀な後輩エンジニア」。指示すれば自分のPC（あなたの環境）でガリガリ作業してくれる。でも後輩に作業を任せる前に、念のため作業前の状態を保存（Git コミット）しておく——というイメージです。

## 2. いま何が起きているのか — 2026年5月の3本柱

2026年5月の Codex アップデートは、ざっくり **3つの方向**に進化しました。すべてアプリ版 **26.527**（公式 changelog のアプリ版エントリ）と、同時期の CLI 更新でまとめて入ってきたものです。

### 柱1. Windows で「コンピュータ操作（Computer Use）」が使えるようになった

これまで Codex の「画面を見て操作する」機能は限られた環境向けでしたが、**Windows でも Computer Use が動くようになりました**。

公式の説明では、Computer Use は **「デスクトップアプリを、見て・クリックして・入力することで操作する」** 機能です（出典: 公式 changelog 26.527）。

つまり、コードエディタの中だけでなく、**Windows 上の普通のアプリ操作そのもの**を Codex に任せられる入り口が開いた、ということです。

### 柱2. スマホ・Mac からの「遠隔操作 / 遠隔確認」

26.527 では **遠隔操作（remote control）が Windows デバイスにも対応**しました。

公式の説明によると、**ChatGPT の iOS / Android アプリ、または Mac から、作業を開始できる**ようになっています（出典: 公式 changelog 26.527）。

「会社のPC（Windows）に作業をさせておいて、外出先のスマホから進み具合を見る」といった使い方が、機能として用意された形です。

### 柱3. 「使用状況・トークン量の見える化」とスレッド管理

26.527 では、地味ですが日常的に効く改善も入りました（出典: 公式 changelog 26.527）。

- **プロフィール欄にユーザー情報・利用統計・トークンの使用状況が表示される**ようになった（どれだけ使っているかが一目で分かる）
- **ローカルのプロジェクトやワークツリーごとに、別々のバックグラウンドスレッドを協調管理**できるようになった
- **過去の Codex アプリのスレッド検索が強化**され、会話の中身や **Git のブランチ名**でも探せるようになった

> 💡 **3本柱を一言で**: 「①Windows でもPCを操作してくれる」「②外からスマホで開始・確認できる」「③どれだけ使ったか・どの作業がどれか、が見やすくなった」。

## 4. Codex が動く場所（4つの入り口＋スマホ遠隔）

Codex は1つのアプリではなく、**同じ Codex をいくつもの入り口から使える**のが特徴です。公式 quickstart が挙げる入り口は **アプリ / IDE拡張 / CLI / クラウドの4種類**。これに加えて、スマホ（ChatGPT モバイルアプリ）からは作業を**遠隔で開始・確認**できます（出典: 公式 quickstart）。

| 場所 | 何か | 対応環境 | こんな人向け |
|---|---|---|---|
| **デスクトップアプリ** | 専用アプリで会話しながら作業 | macOS / Windows | まずはGUIで触りたい人 |
| **IDE 拡張** | エディタの中に Codex が同居 | VS Code / Cursor / Windsurf / VS Code Insiders | 普段のエディタから離れたくない人 |
| **CLI（コマンドライン）** | ターミナルから `codex` で操作 | macOS / Windows / Linux | キーボード中心・自動化したい人 |
| **クラウド** | ブラウザで `chatgpt.com/codex` | Web | PC性能に依存せず使いたい人 |
| **スマホ（遠隔）** | ※独立した入り口ではなく、上の作業を遠隔で開始・確認 | iOS / Android | 外出先で進捗を見たい人 |

### 入り口ごとの始め方（公式の手順）

**デスクトップアプリ（macOS / Windows）**
1. 公式リンクからダウンロード
2. **ChatGPT アカウント**または **OpenAI API キー**でサインイン
3. プロジェクトのフォルダを選ぶ
4. メッセージを送って作業開始

**IDE 拡張**
- VS Code / Cursor / Windsurf / VS Code Insiders に対応。インストールすると **「Codex 拡張が、他の拡張と並んでサイドバーに表示される」**（出典: 公式 quickstart）

**CLI**
```bash
# 標準インストーラ（macOS / Linux）
curl -fsSL https://chatgpt.com/codex/install.sh | sh

# Windows（PowerShell）
powershell -ExecutionPolicy ByPass -c "irm https://chatgpt.com/codex/install.ps1 | iex"

# パッケージマネージャ経由
npm install -g @openai/codex
brew install --cask codex

# 非対話（自動）インストール：環境変数を付けて実行
CODEX_NON_INTERACTIVE=1 curl -fsSL https://chatgpt.com/codex/install.sh | sh
```

**クラウド**
- `chatgpt.com/codex` に直接アクセス。タスクを動かす前に **GitHub 環境連携**をセットアップ（出典: 公式 quickstart）

> 💡 **認証の注意**: サインインは「ChatGPT アカウント」か「OpenAI API キー」のどちらかですが、公式は **「API キー認証だと、クラウドスレッドなど一部の機能が使えないことがある」** と注記しています（出典: 公式 quickstart）。フル機能を使いたいなら ChatGPT アカウントでのサインインが無難です。

## 5. 注目機能の深掘り

### 5-1. Computer Use（コンピュータ操作）

Codex が **画面を見て、クリックして、文字を入力する**ことで、デスクトップアプリを直接操作する機能です。2026年5月の 26.527 で **Windows に対応**しました（出典: 公式 changelog 26.527）。

「コードを書く」だけでなく「アプリを操作する」ところまで広がったのが大きな転換点です。たとえば、

- 表計算アプリを開いてデータを入力する
- ブラウザで情報を集めて、その結果を別アプリに反映する

といった、**コード以外のPC作業**まで任せられる土台ができました。

> ⚠️ 具体的に「どのアプリまで・どこまで安定して操作できるか」は環境差があり、本資料では断定しません。実際に試して挙動を確認してください。

### 5-2. ゴールモード（Goal mode）

「最終的に達成したいゴール」を与えると、Codex がそこに向かって作業を進めるモードです。2026年5月21日の更新で **実験段階を卒業し、アプリ・IDE拡張・CLI のすべてで使えるように**なりました。CLI 側では **デフォルトで有効化**され、専用の保存領域と進捗トラッキングが付きました（出典: 公式 changelog 26.519 / CLI 0.133.0）。

> 💡 ゴールモードは「手順を1つずつ指示する」のではなく「ゴールだけ渡して、道筋は任せる」発想。長めの作業に向いています。

### 5-3. Appshots（アプリのスクショ連携・macOS）

macOS で **両方の Command キーを押すと、最前面のアプリウィンドウをスクリーンショット付きで Codex に送れる**機能です（出典: 公式 changelog 26.519、原文ママの引用を和訳）。

「今見ている画面、これについて聞きたい」を最短で渡せるショートカット、という位置づけです。

### 5-4. MCP / 外部・社内システム連携

Codex は **MCP（Model Context Protocol）** という仕組みで、外部のツールやデータソースと接続できます。5月の更新でも、リモートのワークフローやプラグイン連携が継続的に強化されています（出典: 公式 changelog 各回）。

告知では「**社内に置いたままの MCP サーバーを、外部に公開せずに ChatGPT / Codex / Responses API から安全に使う**」というユースケースが紹介されています。一般的な考え方としては「外向きの通信だけで橋渡しし、社内サーバーを外部公開しない」方向ですが、

> ⚠️ **その接続の具体的な仕組み（tunnel-client を社内側で実行し、JSON-RPC を転送する…といった実装の詳細）は、出典がX上の告知投稿で、本資料では本文を直接取得できませんでした。詳細は未確認**として扱います。実装を検討する場合は公式ドキュメントを確認してください。

### 5-5. プラグイン / マーケットプレイス・権限プロファイル

- **プラグインのマーケットプレイス**経由での共有・発見が、5月を通して整備されました（CLI のマーケットプレイス対応一覧、ChatGPT Business 向けのプラグイン共有など。出典: 公式 changelog 26.519 / CLI 0.131.0・0.133.0）。
- **権限プロファイル（permission profiles）** は、「Codex に何を許可するか」を名前付きで管理する仕組み。一覧API・継承・管理要件のサポートが追加され、`/permissions` から名前付きプロファイルを扱えるようになりました（出典: 公式 changelog CLI 0.133.0・0.135.0）。

### 5-6. バックグラウンドエージェントの識別・スレッド管理

複数の作業を同時に走らせると「どれがどの作業か」が分かりにくくなります。26.527 では **プロジェクト/ワークツリーごとにバックグラウンドスレッドを協調管理**し、**過去スレッドを会話内容や Git ブランチ名で検索**できるようになりました（出典: 公式 changelog 26.527）。

> ⚠️ 告知では「背景エージェントを一目で見分けやすくするアイコン表示の統一」も触れられていますが、UIアイコンの細部はXの告知ベースで、本資料では一次本文を取得できていないため**未確認**として扱います。

## 6. 料金とプラン — どう使い始めるか

公式 quickstart は **「すべての ChatGPT プランに Codex が含まれる（Every ChatGPT plan includes Codex）」** と明記しています。さらに、**OpenAI API キーでサインインすれば API クレジットでも使える**とされています（出典: 公式 quickstart）。

- **手軽に始めたい個人** → 加入している ChatGPT プランでそのままサインインすればOK
- **APIで組み込みたい / 自動化したい** → OpenAI API クレジットを使う（ただし API キー認証では、クラウドスレッドなど一部機能に制限がある場合あり）

### プランの目安（※価格は二次情報・変動します）

各種まとめ・ヘルプセンター情報によると、2026年5月時点の ChatGPT 有料プランはおおむね以下の構成です。**正確な金額・利用上限は変動するため、料金は必ず公式の料金ページで確認してください。**

| プラン | 月額の目安 | 位置づけ |
|---|---|---|
| Free | 0ドル | まず試す |
| Go | 約8ドル | ライト利用 |
| Plus | 約20ドル | 個人の標準 |
| Pro | 約100〜200ドル | ヘビーユーザー（上位ほど利用枠が大きい） |
| Business | 約25〜30ドル/人 | チーム |
| Enterprise | 個別見積 | 組織 |

> ⚠️ なお、各種情報によると **2026年4月2日に Codex の課金が「メッセージ単位」から「API トークン使用量ベース」に変更された**とされています（二次情報）。利用量に応じた従量計算になっている点に注意。**Codex が使うモデルの正式名称（GPT-5 系の具体的バージョン名など）は出典によって表記がばらつくため、本資料では未確認**として断定しません。

## 7. 使い方【超初心者基準】— 悩み別「こう頼むだけ」

ここがこの資料のいちばん大事なパート。**専門用語をできるだけ使わず**、「こういう時はこう頼めばいい」を表にしました。Codex に話しかけるときは、**普通の日本語の文章でOK**です。

### まず最初の3ステップ

1. **フォルダを選ぶ**（自分のプロジェクトのフォルダを Codex に教える）
2. **作業前に保存**（Git で一度コミットしておく＝戻れるようにする）
3. **やってほしいことを文章で頼む**

### 悩み別「こう頼むだけ」対応表

| こんなとき（悩み） | Codex への頼み方（このまま打てばOK） |
|---|---|
| このコード、何をしてるのか分からない | 「このプロジェクトが何をするものか、初心者向けに説明して」 |
| 動かない・エラーが出る | 「エラーの原因を調べて、最小限の変更で直して」 |
| 簡単なものを試しに作りたい | 「このフォルダに、簡単なスネークゲームを作って」 |
| 名前を一括で変えたい | 「この変数名を◯◯から△△に、全部まとめて変えて」 |
| 何から手をつけるか分からない | 「このプロジェクトの改善点を3つだけ、優先順位つきで教えて」 |
| 作業を最後まで任せたい | （ゴールモードで）「最終的に◯◯が動く状態にして」 |
| 外出先で進捗を見たい | スマホの ChatGPT アプリから、走らせた作業の進行状況を確認 |

> 💡 上記の頼み方の例（「このプロジェクトについて教えて」「このリポジトリにスネークゲームを作って」「最小限・高確度の変更でバグを見つけて直して」）は、**公式 quickstart に載っている使用例**を日本語にしたものです（出典: 公式 quickstart）。

### 初心者がまず気をつけること

- **いきなり大事なファイルで試さない**。まずは練習用のフォルダか、コミット済みの状態で。
- **「最小限の変更で」と添える**と、Codex が大胆に書き換えすぎるのを抑えやすい。
- **結果は必ず自分で確認**。AIの提案がいつも正しいとは限りません。

## 9. 裏技・9割が知らない使い方・10倍賢く使うコツ

ここが「撮影の山場」になるパート。**Codex を“ただのコード生成”で終わらせず、エージェントとして使いこなす**ための勘どころを、公式ドキュメントと X（実ユーザーの運用知見）を出典にまとめます。

> 📌 **断り**: 本セクションの見出しは分かりやすさ重視で「裏技/9割が知らない/10倍」という言葉を使っていますが、**中身は出典の取れた事実のみ**です。効果を保証する表現（必ず速くなる等）は使いません。数字での効果は単一ユーザーの体感が多く、ここでは**質的にだけ**紹介します。

### 🃏 裏技編 — 玄人がやっている“設定”

**① AGENTS.md を置く（これが最大のレバー）**

`AGENTS.md` は、**Codex が作業を始める前に必ず読む「プロジェクトの取扱説明書」**。リポジトリのルートに置いた Markdown ファイルです。公式は **「Codex は作業前に `AGENTS.md` を読む」** と明記しています（出典: 公式 AGENTS.md ガイド）。

何を書くかは公式が推奨を示しています（出典: 公式 best-practices）:

- リポジトリの構成・重要ディレクトリ
- ビルド/テストの実行コマンド
- エンジニアリング規約・PR の期待値
- 禁止パターン・制約
- 「完了」と見なす検証方法

公式の刺さる一言: **「短く正確な `AGENTS.md` は、曖昧なルールを並べた長いファイルより役に立つ」**（出典: 公式 best-practices、原文の和訳）。

置き場所と優先順位（出典: 公式 AGENTS.md ガイド）:

- **グローバル**: `~/.codex/`（全プロジェクト共通。`AGENTS.override.md` → `AGENTS.md` の順で読む）
- **プロジェクト**: Git ルートから現在のフォルダへ向かって各階層を読み、**現在地に近いファイルほど優先**（後から連結されるため上書きになる）
- 合計サイズの上限は `project_doc_max_bytes`（既定 **32KiB**）。`~/.codex/config.toml` で調整可

> ⚠️ 重要な誤解ポイント: **`AGENTS.md` に書いたコマンドを Codex が勝手に実行するわけではありません**。あくまで「判断・計画のための指示」として効きます（出典: 公式 AGENTS.md ガイド）。

X でも AGENTS.md は最頻出トピックです。実ユーザーの知見:
- 自分の声・好み・「やること/やらないこと（Dos & Don'ts）」を明記する（[@thesherlocker](https://x.com/thesherlocker/status/2059472529350402162)）
- 繰り返し起きる問題パターンをルール化して、エージェントの混乱を減らす（[@danielchae79](https://x.com/danielchae79/status/2060207303325098410)）
- ミニマリストな AGENTS.md をテンプレとしてコピーして自律化の起点にする（[@Av1dlive](https://x.com/Av1dlive/status/2058588647088406931)）
- 詳細ルールは別ファイルに分け、Global と Project で同期してトークンを節約しつつ一貫性を保つ（[@u1](https://x.com/u1/status/2060713915458425113)）

**② worktree で“並列作業”する**

Git の worktree は、**同じリポジトリを別フォルダに切り出した独立の作業場**。Codex アプリは worktree をサポートし、**並行する変更を隔離**できます（出典: 公式 app）。公式 best-practices も **「git worktree なしでライブのスレッドを走らせるのは避ける」**を“避けるべき失敗”に挙げています（出典: 公式 best-practices）。つまり「複数タスクを同時に回すなら worktree で分ける」がプロの基本。

**③ スレッドは“プロジェクト単位”ではなく“タスク単位”で分ける**

公式 best-practices が挙げる典型的な失敗の一つが **「1プロジェクト1スレッドにしてしまう（タスクごとに分けない）」**こと（出典: 公式 best-practices）。作業（タスク）ごとにスレッドを立てると、文脈が混ざらず精度が上がります。

**④ 別エージェントにレビューさせる（`/review`）**

Codex には **「コミット前に、別の Codex エージェントにコードレビューさせる」**機能があります。CLI/IDE の `/review` で、ベースブランチや未コミット変更に対して構造化レビューを実行できます（出典: 公式 CLI / best-practices）。

**⑤ 複数モデルで相互レビュー（コミュニティの裏技）**

X では **Codex と Claude Code を相互にレビューさせる**「マルチエージェント／敵対的QA」で、計画と実装の品質を上げる運用が共有されています（[@AlexFinn](https://x.com/AlexFinn/status/2012653446349131953)、コミュニティ手法）。河村さんの環境（Codex × Claude Code 併用）と相性が良い使い方です。

### 🙈 9割が知らない使い方編 — “機能の存在”自体

**⑥ Plan Mode（計画モード）**

Plan Mode を ON にすると、Codex は **ブレストし、不明点を質問し、進め方を提示するだけで、ファイルもコマンドも一切触りません**。`/plan` または **Shift+Tab** で切り替えます（出典: 公式 best-practices）。公式の推奨ワークフローは **「重要な作業は Plan Mode で始め → 質問に答え → 計画が合うまで詰め → Plan Mode を切って実行」**（出典: 公式 prompting）。

X でも「計画作成 → 人間がレビュー → 承認してから実行」を必須化する運用が定番です（[@vijucat](https://x.com/vijucat/status/2059130133097889826)、[@heavenOSK](https://x.com/heavenOSK/status/2048295122543280293)）。

**⑦ Skills（再利用できる手順書）**

Skill は **「やり方を一度書けば、以後のセッションが自動で受け継ぐ」再利用ワークフロー**。`SKILL.md` という Markdown に手順を書き、置き場所で適用範囲が決まります（出典: 公式 skills）:

- **ユーザー共通**: `~/.agents/skills/`
- **リポジトリ単位**: `.agents/skills`（カレント）/ `$REPO_ROOT/.agents/skills`
- 呼び出しは `/skills` または `$` メンション。タスク内容にマッチすれば**自動で選ばれる**（progressive disclosure：初期リストは文脈の約2%／約8,000文字に制限）

X の運用知見: **Skill は欲張らず 4〜5個に絞り、エージェント自身にメンテさせる**と効率的（[@jturntdev](https://x.com/jturntdev/status/2059662730618171426)）。定義のコツ（明確な名前・呼び出し形式・使用条件）も共有されています（[@donvito](https://x.com/donvito/status/2005079045315449043)）。

**⑧ 画像を“渡す/作る”**

CLI でも **スクリーンショットやデザイン指定を添付して、プロンプトと一緒に読ませられます**。さらに **画像の生成・編集を CLI/IDE 内で直接**できます（出典: 公式 CLI）。「このエラー画面なんとかして」とスクショを渡す使い方は、9割が見落としがち。

**⑨ MCP / プラグインで“外部ツール”とつなぐ**

MCP（Model Context Protocol）で、Codex に **Figma・Linear・Jira・GitHub・社内Wiki** などの外部ツールへのアクセスを与えられます（出典: 公式 CLI、Web）。プラグイン経由で「アプリ・Skill・MCPサーバー」をまとめて接続できます（出典: 公式 app）。

> 💡 X 由来の運用原則: **MCP は“賢さ”ではなく“アクセスできる世界”を広げる**もの。常に文脈を消費するので、常時必要でない情報は MCP より Skill 側に寄せると軽い（コミュニティ知見）。

**⑩ 定期実行（Automations）とスクリプト化（`codex exec`）**

アプリの **Automations** は、**繰り返しタスクのスケジュール実行や、同じスレッドを定期的に起こして点検**できます（出典: 公式 app）。CLI の `codex exec` を使えば、**繰り返しのワークフローをスクリプト化**できます（出典: 公式 CLI）。

**⑪ subagents で並列化**

複雑なタスクは **subagents で分担・並列化**できます（出典: 公式 CLI）。大きな作業を分割して同時に進めたいときの土台です。

### 🚀 10倍賢く使うコツ編 — “頼み方”と“運用”

**⑫ 最初のプロンプトは「4点セット」で書く**

公式 best-practices が挙げる、出力品質を最も左右する最初の一手。次の4つを入れるだけで結果が変わります（出典: 公式 best-practices、各項目は原文の和訳）:

| 要素 | 何を書くか |
|---|---|
| **Goal（目的）** | 何を変えたい/作りたいのか |
| **Context（文脈）** | 関係するファイル・フォルダ・ドキュメント・例・エラー |
| **Constraints（制約）** | 守るべき規約・設計・安全要件・慣習 |
| **Done when（完了条件）** | 何が真なら完了か（テストが通る／挙動が変わる／バグが再現しなくなる 等） |

**⑬ 推論レベル（reasoning effort）を使い分ける**

タスクの重さで思考の深さを変えます（出典: 公式 best-practices / IDE）:

- **Low**: 速い・スコープが明確な作業
- **Medium / High**: 複雑な変更やデバッグ
- **Extra High**: 長くてエージェント的な、思考量の多い作業

「軽い修正に High を使って遅い」「重いデバッグに Low で浅い」を避けるだけで体感が変わります。

**⑭ 同じミスを2回したら、AGENTS.md を直す**

公式 best-practices の金言: **「Codex が同じミスを2回したら、振り返り（retrospective）を求めて `AGENTS.md` を更新する」**（出典: 公式 best-practices、原文の和訳）。AGENTS.md を“育てる living document”として運用するのが、長期で効く最大のコツ。

**⑮ いきなり実装させず「先に計画を聞く」**

複雑な作業ほど、Plan mode を使うか、**「先に計画を提案して」「逆に質問して（interview）」**と頼んでから着手させると、手戻りが減ります（出典: 公式 best-practices / prompting）。

**⑯ Codex に“検証まで”やらせる**

公式は **「必要ならテストを作り、関連チェックを実行し、結果を確認し、受け入れる前にレビューさせる」**一連のループを推奨（出典: 公式 best-practices）。「作って終わり」ではなく「テストが通る状態まで」を一度に頼むと品質が上がります。

> 🎬 **撮影用・3行まとめ**:
> 1. まず `AGENTS.md` を1枚置く（短く正確に）＝ これだけで賢くなる
> 2. 重い作業は **Plan Mode** で計画→承認→実行、軽さ/重さで **推論レベル**を切替
> 3. **同じミス2回で AGENTS.md を更新**・**タスク単位でスレッド**・**worktreeで並列** = 玄人運用

## 10. つまずきポイントと注意

- **「動きます」を鵜呑みにしない**: Codex は実際にファイルを書き換えます。作業前の Git コミットを習慣に。
- **API キー認証の制限**: 一部機能（クラウドスレッド等）が使えない場合あり。フル機能なら ChatGPT アカウントでサインイン。
- **アプリ版と CLI 版で番号が別**: 「26.5xx」はアプリ、「0.13x.0」は CLI。アップデート確認時は混同しない。
- **Computer Use / 遠隔操作は権限が強い**: PCを操作したり外部から作業を始められる分、**何を許可しているか**を権限プロファイルで把握しておく。
- **日付・価格・モデル名は変動する**: 本資料の未確認項目は、必ず一次情報で最新を確認。

## 11. まとめ

2026年5月の Codex は、**「コードを書くAI」から「PC作業まで肩代わりするエージェント」への移行**が一段進んだ月でした。

- **Windows で Computer Use 対応**（アプリ 26.527）— PC操作そのものを任せられる
- **スマホ・Mac からの遠隔操作 / 遠隔確認**— 席を立っても作業が進む
- **使用量・トークンの見える化、スレッド管理の改善**— 複数作業を回しやすく
- **CLI/SDK の継続強化**— `codex doctor`、ゴールモード正式化、Appshots、プラグイン/マーケットプレイス、権限プロファイル、構造化出力

「まず触ってみる人」は **アプリか IDE 拡張**から、「自動化したい人」は **CLI / Python SDK** から。どちらにせよ、**作業前の Git コミット**と**最小限の変更で頼む**の2つを守れば、安全に試し始められます。

## 13. よくある質問（FAQ）

**Q. プログラミングが分からなくても使えますか？**
A. 「読む・調べる・小さく試す」用途なら、普通の日本語で頼めます（例:「このプロジェクトが何をするものか説明して」）。ただし Codex は実際にファイルを変更できるので、**作業前の保存（Git コミット）**と**結果の自分での確認**は必要です。

**Q. Mac と Windows、どちらでも使えますか？**
A. デスクトップアプリは macOS / Windows、CLI は macOS / Windows / Linux に対応しています（出典: 公式 quickstart）。さらに 2026年5月の 26.527 で、**Windows でも Computer Use（PC操作）**が使えるようになりました（出典: 公式 changelog 26.527）。

**Q. スマホだけで使えますか？**

**Q. どの入り口から始めるのがいいですか？**
A. **GUIで触りたいならアプリ**、**普段のエディタから離れたくないなら IDE 拡張**、**自動化したいなら CLI / Python SDK** が目安です。迷ったら、まずアプリか IDE 拡張から始めると分かりやすいです。
